## 三、性能分析

### 3.1 延迟趋势
- **P50 延迟稳定**：30路/60路/100路并发下 P50 分别为 518ms / 722ms / 726ms，表现稳定
- **P99 随并发增长**：10,443ms → 13,387ms → 16,551ms，高并发下长尾延迟明显
- **超长延迟异常**：100路并发时出现 60s 超长延迟，可能是 LLM 服务响应慢或网络抖动

### 3.2 吞吐量趋势
- 30路并发：~11.7 req/s
- 60路并发：~40.6 req/s
- 100路并发：~47.3 req/s
- **分析**：60路到100路吞吐量增长放缓，接近系统瓶颈

### 3.3 Token 统计问题
- **采集率低**：30路仅采集12个 token 数据点（40%），60路仅10个（17%），100路仅11个（11%）
- **原因分析**：LLM 流式响应中 `usage` 字段仅在最后一个 chunk 返回，部分请求流结束时未正确解析

## 四、Bad Case 分析

### Bad Case 1: 长上下文裁剪不当

**问题分析**：
1. 当前实现中的上下文管理采用简单的 FIFO 队列方式，没有区分信息的重要性
2. 历史消息被均匀裁剪，可能丢失用户当前关心的重要上下文
3. 在高并发场景下，多个会话同时累积长上下文，导致 LLM 服务端处理压力增大

**修复思路**：
1. **智能裁剪策略**：实现基于语义相关性的上下文裁剪，优先保留与当前问题相关的历史对话，而非简单的按时间顺序裁剪
2. **摘要机制**：对早期的历史对话进行摘要压缩，保留关键信息而非完整原文
3. **分层存储**：将长期记忆（用户偏好、重要设定）和短期对话上下文分开管理，避免重复传输无关信息
4. **动态阈值**：根据当前查询的复杂度动态调整上下文长度，简单查询使用更短的上下文以提升响应速度

---

### Bad Case 2: 工具调用超时

**问题分析**：
1. 工具执行（如数据库查询、外部 API 调用）没有独立的超时控制，可能被 LLM 的整体超时机制影响
2. 部分工具（如查询账单）可能涉及大量数据扫描，在高并发下数据库响应变慢
3. 工具调用的重试机制缺失或配置不当，网络抖动时无法自动恢复

**修复思路**：
1. **分级超时控制**：为不同类型的工具设置独立的超时时间，如数据库查询 3s、外部 API 调用 5s，确保单一工具超时不影响整体流程
2. **异步工具执行**：将工具调用改为异步模式，工具执行期间可以继续处理其他请求，提升并发能力
3. **降级策略**：当工具超时时，提供合理的降级响应（如"暂时无法获取数据，请稍后重试"），而不是让整个请求失败
4. **缓存机制**：对频繁查询且数据变化不大的工具结果（如预算查询）增加缓存，减少重复调用

---

### Bad Case 3: Memory 误写入

**问题分析**：
1. 理论上不会出现不同用户memory混乱的问题
2. 如果Memory 写入逻辑中缺少用户身份的强校验，在高并发场景下可能出现并发竞争导致的数据混乱
2. 自动提取 Memory 的规则不够精确，将临时性的对话内容误判为需要长期记忆的信息
3. Memory 写入缺少验证机制，没有对提取的内容进行用户确认或置信度评估

**修复思路**：
1. **用户隔离强化**：在 Memory 写入前增加用户身份的二次校验，确保每个用户的 Memory 严格隔离，使用用户 ID 作为分片键
2. **写入前确认**：对于自动提取的 Memory 内容，通过置信度评分机制，低置信度的内容需要用户确认后再写入
3. **分类存储**：将 Memory 分为"用户确认"和"系统提取"两类，系统提取的内容在后续对话中可作为参考但不影响核心判断
4. **定期清理**：实现 Memory 的过期和清理机制，对于长时间未被引用或用户已否定的内容自动清理，避免错误累积

---

## 五、总结与建议

| 问题 | 建议 |
|------|------|
| 高并发长尾延迟 | 缩短超时时间，增加熔断机制，优化 LLM 调用链路 |
| Token 采集率低 | 增加响应头传递 token，改进流解析逻辑，提高采集成功率 |
| 长上下文裁剪不当 | 实现智能裁剪策略，引入摘要机制，分层存储上下文 |
| 工具调用超时 | 分级超时控制，异步执行，增加降级和缓存策略 |
| Memory 误写入 | 强化用户隔离，增加写入确认机制，分类存储并定期清理 |
