### 1.2 组件职责

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| **kapi-server** | 主服务，路由分发、流式响应、会话管理 | `cmd/server`, gorilla/mux |
| **Trace Middleware** | 提取/生成 X-Trace-ID，注入 context，全链路日志串联 | `internal/logger` |
| **Agent Runtime** | Function Calling 编排循环，驱动 LLM ↔ skill-server 交互 | `internal/agent` |
| **Memory Service** | 用户记忆读写，3 层存储（Redis→MySQL→空），Token 预算裁剪 | `internal/memory` |
| **Memory Determiner** | 每轮对话后异步调用 LLM 提取 profile/facts/summary，写回 Memory | `internal/memory/determiner.go` |
| **Session Manager** | 会话状态管理，Redis 存储，TTL 自动过期 | `internal/session` |
| **skill-server** | 独立 HTTP 服务，Skill 声明加载、页面裁剪、工具执行 | `cmd/skill-server` |
| **Skill Loader** | 从磁盘 JSON 加载 SkillConfig，构建 tool 索引 | `internal/skillserver/loader.go` |
| **Executor** | 工具实际执行层（add_bill / query_bills / budget），支持 mock 降级 | `internal/skillserver/executor.go` |
| **存储层** | Redis（session/memory 热缓存）+ MySQL（memory/bills/budgets 持久化）| `internal/storage`, `internal/repository` |

### 2.3 skill-server 独立部署（热更新路径）

```
开发者          skill-server 集群        Load Balancer    kapi-server
  │                    │                       │               │
  │─修改 JSON/Go────>  │                       │               │
  │─重新构建镜像──────>  │                       │               │
  │                    │                       │               │
  │─滚动更新 pod-2─────>│                       │               │
  │                    │  pod-1(healthy)        │               │
  │                    │  pod-2(starting)       │─自动摘流─>    │
  │                    │  pod-3(healthy)        │               │
  │                    │                        │               │
  │                    │  pod-2(ready /health)  │               │
  │                    │                        │─恢复流量──>   │
  │                    │  pod-1(rolling update)─│               │
  │                    │                        │               │
```

> **设计说明：** skill-server 无状态，kapi-server 每次请求动态调用 `GET /skills`，不缓存 skill 列表。skill-server 滚动重启期间 LB 自动摘流，kapi-server 侧即使短暂 503 也有 fallback（`proceeding without tools`）。

## 3. 数据模型

### 3.1 Session Schema

```go
type Session struct {
    ID          string                   `json:"id"`           // sess_1234567890
    UserID      string                   `json:"user_id"`
    Messages    []map[string]interface{} `json:"messages"`     // 含 role/content/timestamp
    CreatedAt   string                   `json:"created_at"`   // RFC3339
    LastUpdated string                   `json:"last_updated"`
    ExpiresAt   string                   `json:"expires_at"`
}
```

### 3.2 Message Schema

```go
type Message struct {
    Role      string      `json:"role"`       // user | assistant | tool
    Content   string      `json:"content"`
    ToolCalls []ToolCall  `json:"tool_calls,omitempty"`
    Timestamp time.Time   `json:"timestamp"`
    Metadata  interface{} `json:"metadata,omitempty"`
}
```

### 3.3 Memory Schema

```go
// Redis key: "memory:{user_id}"，TTL 24h
// MySQL table: user_memory（一用户一行，profile 快照模型）
type Memory struct {
    UserID        string   `json:"user_id"`
    Profile       string   `json:"profile"`        // 长期画像，一段话描述
    Preferences   string   `json:"preferences"`    // 交互偏好，一段话描述
    Facts         []string `json:"facts"`          // 关键事实，上限 30 条，去重
    RecentSummary string   `json:"recent_summary"` // 近期对话滚动摘要，上限 500 tokens
    UpdatedAt     string   `json:"updated_at"`
}
```

**三层存储策略：**
- **热**：Redis，TTL 24h，读写优先
- **温**：MySQL `user_memory` 表，Redis miss 时回填并预热缓存
- **冷**：首次用户，创建空 Memory 对象

**Facts 写入规则：** 大小写不敏感去重，超过 30 条时淘汰最旧一条（FIFO）。

### 3.4 Skill Schema

```go
// 声明式定义，存放于 configs/*.json
type SkillConfig struct {
    ID             string     `json:"id"`
    Name           string     `json:"name"`
    Description    string     `json:"description"`
    Layer          string     `json:"layer"`           // built-in | org | user
    SupportedPages []string   `json:"supported_pages"` // 空 = 全局，支持 "*" 通配
    Tools          []ToolSpec `json:"tools"`
}

type ToolSpec struct {
    Name           string                 `json:"name"`
    Description    string                 `json:"description"`
    Parameters     map[string]interface{} `json:"parameters"`      // OpenAI function schema
    ActionType     ActionType             `json:"action_type"`     // return_direct | llm_summary | next_step
    ReturnTemplate string                 `json:"return_template"` // return_direct 时的话术模板
    NextTool       string                 `json:"next_tool"`       // next_step 时的下一个工具
}
```

**ActionType 语义：**
| 值 | 场景 | 行为 |
|---|---|---|
| `return_direct` | 操作类（add_bill） | 用 ReturnTemplate 渲染结果直接返回用户，终止循环 |
| `llm_summary` | 查询类（query_bills） | 将原始结果回灌 LLM，由 LLM 生成自然语言回复 |
| `next_step` | 多步编排 | 执行完当前工具后强制串行执行 NextTool |

### 3.5 API 路由表

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/sessions` | 创建会话 |
| GET | `/sessions/{id}` | 获取会话 |
| DELETE | `/sessions/{id}` | 删除会话 |
| POST | `/sessions/{id}/messages` | 发送消息（非流式） |
| POST | `/sessions/{id}/stream` | 发送消息（SSE 流式） |
| GET | `/memory?user_id=` | 获取用户 Memory |
| POST | `/memory?user_id=` | 更新 Memory（profile/preferences/fact） |
| DELETE | `/memory?user_id=` | 清空用户全部 Memory |
| DELETE | `/memory/facts/{index}?user_id=` | 删除指定事实（用户主动遗忘） |
| GET | `/health` | 健康检查 |

## 4. Trace 与日志

### 4.1 Trace ID 传播链路

```
Client ─HTTP Header: X-Trace-ID──> kapi-server
                                       │ context.WithValue
                                       ├──> Agent Runtime
                                       ├──> Memory Service
                                       └──> skill-server（Header: X-Trace-ID 透传）
                                                │ context.WithValue
                                                └──> Executor
```

- 每个 HTTP 请求在 `traceMiddleware` 提取或生成 16 位 hex Trace ID
- 通过 Go context 在进程内传播，通过 `X-Trace-ID` HTTP header 跨服务传播
- 所有日志通过 `logger.FromContext(ctx)` 自动携带 `trace_id` 字段

### 4.2 日志格式

```
time="2026-04-25 22:00:01.123" level=info file="internal/agent/agent.go:194"
  trace_id=a1b2c3d4e5f6g7h8 session_id=sess_xxx user_id=user_001
  msg="[agent] loop started"
```

- 格式：`logrus TextFormatter`，含绝对时间戳、文件:行号、结构化字段
- 级别：Debug（chunk 级流式日志）/ Info（关键节点）/ Warn（可恢复错误）/ Error（需关注）

## 5. 并发与状态管理

### 5.1 会话处理

- 单会话串行处理（请求串行排队），保证消息顺序
- 多会话并发，Redis 隔离，无跨会话共享状态
- HTTP Client 超时：Agent→LLM 90s（适配推理模型），Agent→skill-server 30s

### 5.2 限流策略

| 维度 | 限制 | 实现 |
|------|------|------|
| IP | 100 req/min | 内存 map + 滑动窗口 |

### 5.3 并发扩容路径

```
阶段1：当前（单机）
├── kapi-server × 1
├── skill-server × 1
├── Redis 单节点
└── MySQL 单节点

阶段2：灰度
├── kapi-server × 2 + Nginx LB
├── skill-server × 2（无状态，独立滚动更新）
├── Redis Sentinel
└── MySQL 读写分离

阶段3：生产
├── kapi-server × 3 + K8s HPA
├── skill-server × 3（readinessProbe: GET /health）
├── Redis Cluster (3主3从)
└── MySQL 主从集群
```

## 6. 延迟预算分解

### 6.1 端到端延迟目标

| 指标 | 目标 | P95 |
|------|------|-----|
| 首 token 延迟 | ≤ 1.5s | 2s |
| 整轮响应（纯对话） | ≤ 4s | 5s |
| 整轮响应（含工具调用） | ≤ 6s | 8s |

### 6.2 延迟分解

```
纯对话 5s
├── 中间件（trace+log+cors+限流）: 10ms
├── Session 加载（Redis）: 20ms
├── Memory 读取（Redis 命中）: 20ms
├── Prompt 组装 + TrimToBudget: 5ms
├── LLM 调用（流式）: ~3500ms
│   ├── 推理模型 reasoning phase: ~2000ms
│   └── 生成 + 网络传输: ~1500ms
└── Session 保存: 20ms

含工具调用额外
├── GET /skills: ~10ms
├── POST /execute（DB 操作）: ~50ms
└── 第二轮 LLM 调用（llm_summary）: ~2000ms
```

### 6.3 Memory Determiner 异步化

Memory 写入（MemoryDeterminer LLM 调用，~3s）在响应完成后异步执行，不在关键路径上，不影响用户感知延迟。

## 7. 容量与成本估算

### 7.1 1万DAU场景

**假设：** 日活 10,000，日均对话 3 次，平均 5 轮，平均 200 输入 + 400 输出 tokens

```
日 Token = 10,000 × 3 × 5 × 600 = 90M tokens
月 Token = 90M × 30 = 2.7B tokens
```

### 7.2 模型月成本（CNY）

| 方案 | 主模型 | Token 成本 | 基础设施 | 月合计 |
|------|--------|-----------|----------|--------|
| **推荐** | DeepSeek V3 | ¥4,050 | ¥200 | **¥4,250** |
| 低成本 | GLM-4-Flash | ¥3,375 | ¥200 | **¥3,575** |

**分级策略（推荐）：**
- 简单对话 40% → GLM-4-Flash（低成本快响应）
- 工具调用/复杂推理 60% → DeepSeek V3
- 综合月成本约 ¥3,035 ≈ $422

## 8. 故障预案

| 故障 | 降级策略 |
|------|----------|
| **skill-server 不可用** | Agent 继续运行，无工具调用，纯 LLM 对话（已实现 fallback log） |
| **LLM 超时/限流** | 返回 mock 关键词匹配回复（`getMockResponse`） |
| **Redis 不可用** | Memory 降级：直接读 MySQL；Session 降级：空历史 |
| **MySQL 不可用** | Memory 仅 Redis 存储，写入跳过 memSaveCallback（Warn 日志） |
| **MemoryDeterminer 超时** | goroutine 60s context 超时，Warn 日志，本轮 Memory 不更新，不影响主流程 |

### 8.1 降级等级

| 等级 | 触发条件 | 影响 |
|------|----------|------|
| L0 | 正常 | 完整功能 |
| L1 | Redis 延迟 >100ms | Memory 读 MySQL，轻微延迟增加 |
| L2 | Redis 不可用 | 无 session 历史，Memory 读 MySQL |
| L3 | skill-server 不可用 | 无工具调用，纯对话模式 |
| L4 | LLM 超时/限流 | Mock 关键词回复兜底 |
| L5 | MySQL 不可用 | Memory 仅内存，会话结束后丢失 |

## 9. 安全设计

### 9.1 输入安全

- System Prompt 与 User Message 严格分离（OpenAI messages 格式）
- 工具参数类型校验在 executor 层强制执行
- 金额必须 > 0，category 受 enum 约束

### 9.2 PII 脱敏

| 数据类型 | 方式 |
|---------|------|
| 手机号 | 前3后4，中间 `****` |
| 邮箱 | 用户名部分打码 |
| 银行卡 | 前6后4 |
