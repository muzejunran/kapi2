### 1.2 组件职责

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| **kapi-server** | 主服务，路由分发、流式响应、会话管理 | `cmd/server`, gorilla/mux |
| **Trace Middleware** | 提取/生成 X-Trace-ID，注入 context，全链路日志串联 | `internal/logger` |
| **Agent Runtime** | Function Calling 编排循环，驱动 LLM ↔ skill-server 交互 | `internal/agent` |
| **Memory Service** | 用户记忆读写，3 层存储（Redis→MySQL→空），Token 预算裁剪 | `internal/memory` |
| **Memory Determiner** | 每轮对话后异步调用 LLM 提取 profile/facts/summary，写回 Memory | `internal/memory/determiner.go` |
| **Session Manager** | 会话状态管理，Redis 存储，TTL 自动过期 | `internal/session` |
| **skill-server** | 独立 HTTP 服务，Skill 声明加载、页面裁剪、工具执行 | `cmd/skill-server` |
| **Skill Loader** | 从磁盘 JSON 加载 SkillConfig 构建 tool 索引；加载 `pages.json` 构建 PageIndex（页面→skill ID 映射） | `internal/skillserver/loader.go` |
| **Executor** | 工具实际执行层（add_bill / query_bills / budget），支持 mock 降级 | `internal/skillserver/executor.go` |
| **存储层** | Redis（session/memory 热缓存）+ MySQL（memory/bills/budgets 持久化）| `internal/storage`, `internal/repository` |

### 2.2 含工具调用的对话（Function Calling 循环）

```
Client    kapi-server    Agent Runtime    skill-server    LLM    MySQL
  │            │               │               │           │        │
  │─POST stream───────────────>│               │           │        │
  │            │               │─GET /skills?page_context─>│        │
  │            │               │<─[{type:function,...}]────│        │
  │            │               │                           │        │
  │            │               │  [注入 system prompt 工具约束]      │
  │            │               │  "当前页面支持: add_bill, ..."      │
  │            │               │                           │        │
  │            │               │─POST /chat(messages+tools, stream=false)──>│
  │            │               │                           │        │
  │            │  ┌── LLM 选中工具 ──────────────────────────────────┤
  │            │  │            │<─{tool_call:{name,args}}──────────│
  │<─SSE:step("正在记录账单...")│               │           │        │
  │            │               │─POST /execute─────────────>        │
  │            │               │  {tool_name,user_id,args} │        │
  │            │               │               │─INSERT────────────>│
  │            │               │               │<─ok───────────────│
  │            │               │<─{success,action_type,result}──────│
  │            │               │                           │        │
  │            │  (action_type=return_direct)              │        │
  │<─SSE:text("已为您记录：餐饮 45元")         │           │        │
  │<─SSE:done──────────────────│               │           │        │
  │            │  (action_type=llm_summary)    │           │        │
  │            │               │─POST /chat(messages+tool_result, stream=true)──>│
  │<─SSE:text(LLM 自然语言总结)─────────────────────────────────────│
  │<─SSE:done──────────────────│               │           │        │
  │            │               │               │           │        │
  │            │  └── LLM 未选中任何工具 ─────────────────────────────┤
  │            │               │<─{content:"当前页面仅支持..."}──────│
  │<─SSE:text(拒绝文案，说明当前页面能做什么)  │           │        │
  │<─SSE:done──────────────────│               │           │        │
```

> **设计说明：** 带 tools 的第一轮 LLM 调用使用 `stream=false`（blocking），避免将工具选择阶段的中间文本推送给前端。system prompt 中注入当前页面支持的 tool 名称和描述，引导 LLM 在无法匹配工具时给出具体的能力边界说明，而不是自由回答。

### 2.3 模型路由策略

> **当前状态：** Demo 中所有 LLM 调用（工具选择、工具结果总结、普通对话、Memory Determiner）共用同一个模型，由 `MODEL_NAME` 环境变量统一配置。

不同调用阶段对模型能力的要求差异显著，生产环境建议按场景分级使用不同模型：

| 调用场景 | 特点 | 推荐模型级别 |
|---------|------|------------|
| **工具选择**（blocking，第一轮） | 快速意图识别，无需长文本生成 | 轻量模型（GLM-4-Flash） |
| **工具结果总结**（llm_summary，stream） | 自然语言组织，中等复杂度 | 中等模型（DeepSeek V3） |
| **普通对话**（无 tools，stream） | 上下文理解，流畅生成 | 中等模型 |
| **Memory Determiner**（异步） | 结构化信息抽取，非关键路径 | 轻量模型 |

工具选择阶段用轻量模型收益最大：该步骤是 blocking 调用，直接影响首个 StepEvent 的延迟，且任务简单（理解意图 + 选工具），不需要强推理能力。

**兜底切换机制（演进方向）：**

当前已有 `getMockResponse` 作为 LLM 完全不可用时的关键词兜底。生产建议构建模型级 fallback chain：

```
主模型（DeepSeek V3）
  └─ 超时/限流/5xx ──> 备用模型（GLM-4-Flash）
                          └─ 也失败 ──> getMockResponse（关键词兜底）
```

每次模型切换记录 Warn 日志（含原因和切换目标），便于告警触发和计费核查。

### 2.4 skill-server 独立部署（热更新路径）

```
开发者          skill-server 集群        Load Balancer    kapi-server
  │                    │                       │               │
  │─修改 JSON/Go────>  │                       │               │
  │─重新构建镜像──────>  │                       │               │
  │                    │                       │               │
  │─滚动更新 pod-2─────>│                       │               │
  │                    │  pod-1(healthy)        │               │
  │                    │  pod-2(starting)       │─自动摘流─>    │
  │                    │  pod-3(healthy)        │               │
  │                    │                        │               │
  │                    │  pod-2(ready /health)  │               │
  │                    │                        │─恢复流量──>   │
  │                    │  pod-1(rolling update)─│               │
  │                    │                        │               │
```

> **设计说明：** skill-server 无状态，kapi-server 每次请求动态调用 `GET /skills`，不缓存 skill 列表。skill-server 滚动重启期间 LB 自动摘流，kapi-server 侧即使短暂 503 也有 fallback（`proceeding without tools`）。

## 3. 数据模型

### 3.1 Session Schema

```go
type Session struct {
    ID          string                   `json:"id"`           // sess_1234567890
    UserID      string                   `json:"user_id"`
    Messages    []map[string]interface{} `json:"messages"`     // 含 role/content/timestamp
    CreatedAt   string                   `json:"created_at"`   // RFC3339
    LastUpdated string                   `json:"last_updated"`
    ExpiresAt   string                   `json:"expires_at"`
}
```

### 3.2 Message Schema

```go
type Message struct {
    Role      string      `json:"role"`       // user | assistant | tool
    Content   string      `json:"content"`
    ToolCalls []ToolCall  `json:"tool_calls,omitempty"`
    Timestamp time.Time   `json:"timestamp"`
    Metadata  interface{} `json:"metadata,omitempty"`
}
```

### 3.3 Memory Schema

```go
// Redis key: "memory:{user_id}"，TTL 24h
// MySQL table: user_memory（一用户一行，profile 快照模型）
type Memory struct {
    UserID        string   `json:"user_id"`
    Profile       string   `json:"profile"`        // 长期画像，一段话描述
    Preferences   string   `json:"preferences"`    // 交互偏好，一段话描述
    Facts         []string `json:"facts"`          // 关键事实，上限 30 条，去重
    RecentSummary string   `json:"recent_summary"` // 近期对话滚动摘要，上限 500 tokens
    UpdatedAt     string   `json:"updated_at"`
}
```

**三层存储策略：**
- **热**：Redis，TTL 24h，读写优先
- **温**：MySQL `user_memory` 表，Redis miss 时回填并预热缓存
- **冷**：首次用户，创建空 Memory 对象

**Facts 写入规则：** 大小写不敏感去重，超过 30 条时淘汰最旧一条（FIFO）。

### 3.4 Skill Schema

```go
// 声明式定义，存放于 cmd/skill-server/configs/*.json
type SkillConfig struct {
    ID    string     `json:"id"`
    Name  string     `json:"name"`
    Layer string     `json:"layer"`  // built-in | org | user
    Tools []ToolSpec `json:"tools"`
}

type ToolSpec struct {
    Name           string                 `json:"name"`
    Description    string                 `json:"description"`
    Parameters     map[string]interface{} `json:"parameters"`      // OpenAI function schema
    ActionType     ActionType             `json:"action_type"`     // return_direct | llm_summary | next_step
    ReturnTemplate string                 `json:"return_template"` // return_direct 时的话术模板
    NextTool       string                 `json:"next_tool"`       // next_step 时的下一个工具
}

// 页面路由：集中配置，存放于 cmd/skill-server/configs/pages.json
// key 为页面标识，value 为该页面允许的 skill ID 列表；"*" 表示全局可用
type PageIndex map[string][]string
```

**页面路由设计：**

页面与 skill 的映射集中在 `pages.json` 中，skill 配置本身不感知页面：

```json
{
  "*":             ["web_search"],
  "home":          ["add_bill", "query_bills"],
  "bills.detail":  ["query_bills", "add_bill"],
  "bills.add":     ["add_bill"],
  "budget.index":  ["budget_advisor"]
}
```

- 新增页面：只改 `pages.json`，所有 skill 配置不动
- 新增 skill：部署 skill 的 JSON config，再在 `pages.json` 中将其加入对应页面
- `"*"` key 下的 skill 在所有页面均可用（全局 skill）

**ActionType 语义：**
| 值 | 场景 | 行为 |
|---|---|---|
| `return_direct` | 操作类（add_bill） | 用 ReturnTemplate 渲染结果直接返回用户，终止循环 |
| `llm_summary` | 查询类（query_bills） | 将原始结果回灌 LLM，由 LLM 生成自然语言回复 |
| `next_step` | 多步编排 | 执行完当前工具后强制串行执行 NextTool |

### 3.5 API 路由表

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/sessions` | 创建会话 |
| GET | `/sessions/{id}` | 获取会话 |

## 4. Trace 与日志

### 4.1 Trace ID 传播链路

```
Client ─HTTP Header: X-Trace-ID──> kapi-server
                                       │ context.WithValue
                                       ├──> Agent Runtime
                                       ├──> Memory Service
                                       └──> skill-server（Header: X-Trace-ID 透传）
                                                │ context.WithValue
                                                └──> Executor
```

- 每个 HTTP 请求在 `traceMiddleware` 提取或生成 16 位 hex Trace ID
- 通过 Go context 在进程内传播，通过 `X-Trace-ID` HTTP header 跨服务传播
- 所有日志通过 `logger.FromContext(ctx)` 自动携带 `trace_id` 字段

### 4.2 日志格式

```
time="2026-04-25 22:00:01.123" level=info file="internal/agent/agent.go:194"
  trace_id=a1b2c3d4e5f6g7h8 session_id=sess_xxx user_id=user_001
  msg="[agent] loop started"
```

- 格式：`logrus TextFormatter`，含绝对时间戳、文件:行号、结构化字段
- 级别：Debug（chunk 级流式日志）/ Info（关键节点）/ Warn（可恢复错误）/ Error（需关注）

## 5. 并发与状态管理

### 5.1 会话处理

- 单会话串行处理（请求串行排队），保证消息顺序
- 多会话并发，Redis 隔离，无跨会话共享状态
- HTTP Client 超时：Agent→LLM 90s（适配推理模型），Agent→skill-server 30s

### 5.2 限流策略

| 维度 | 限制 | 实现 |
|------|------|------|
| IP | 100 req/min | 内存 map + 滑动窗口 |

### 5.3 并发扩容路径

## 6. 延迟预算分解

### 6.1 端到端延迟目标

| 指标 | 目标 | P95 |
|------|------|-----|
| 首 token 延迟 | ≤ 1.5s | 2s |
| 整轮响应（纯对话） | ≤ 4s | 5s |
| 整轮响应（含工具调用） | ≤ 6s | 8s |

### 6.2 延迟分解

```
纯对话 5s
├── 中间件（trace+log+cors+限流）: 10ms
├── Session 加载（Redis）: 20ms
├── Memory 读取（Redis 命中）: 20ms
├── Prompt 组装 + TrimToBudget: 5ms
├── LLM 调用（流式）: ~3500ms
│   ├── 推理模型 reasoning phase: ~2000ms
│   └── 生成 + 网络传输: ~1500ms
└── Session 保存: 20ms

含工具调用额外
├── GET /skills: ~10ms
├── LLM blocking 调用（工具选择，stream=false）: ~500ms～1s
├── POST /execute（DB 操作）: ~50ms
└── 第二轮 LLM 调用（llm_summary，stream=true）: ~2000ms
```

### 6.3 Memory Determiner 异步化

Memory 写入（MemoryDeterminer LLM 调用，~3s）在响应完成后异步执行，不在关键路径上，不影响用户感知延迟。

## 7. 容量与成本估算

### 7.1 1万DAU场景

**假设：** 日活 10,000，日均对话 3 次，平均 5 轮，平均 200 输入 + 400 输出 tokens

```
日 Token = 10,000 × 3 × 5 × 600 = 90M tokens
月 Token = 90M × 30 = 2.7B tokens
```

### 7.2 模型月成本（CNY）

| 方案 | 主模型 | Token 成本 | 基础设施 | 月合计 |
|------|--------|-----------|----------|--------|
| **推荐** | DeepSeek V3 | ¥4,050 | ¥200 | **¥4,250** |
| 低成本 | GLM-4-Flash | ¥3,375 | ¥200 | **¥3,575** |

**分级策略（推荐，见 2.3 模型路由策略）：**
- 工具选择 + Memory Determiner → GLM-4-Flash（轻量，约占调用量 50%）
- 工具结果总结 + 普通对话 → DeepSeek V3（中等，约占 50%）
- 综合月成本约 ¥3,035 ≈ $422

## 8. 故障预案

> **当前状态：** Demo 中绝大多数失败路径直接降级为 mock 回复或 Warn 日志，适合本地开发验证。以下描述**生产目标**的完整处理策略，标注 `[未实现]` 的为演进方向。

### 8.1 LLM 调用失败

LLM 是核心链路，不能以 mock 兜底作为唯一手段，应构建多级 fallback：

```
主模型（如 DeepSeek V3）
  └─ 超时 / 限流 / 5xx
       └─ 备用模型（如 GLM-4-Flash）[未实现]
             └─ 也失败
                  └─ getMockResponse（关键词匹配兜底，已实现）
```

- 主→备切换触发条件：HTTP 5xx、连接超时（>10s）、rate limit 429
- 每次切换写 Warn 日志（含原因、目标模型），用于告警和计费核查
- skill-server 不可用：Agent 继续运行，无工具调用，降级为纯 LLM 对话（已实现）

### 8.2 存储写入失败

**用户消息不能丢失**，写入失败不能仅打 Warn 日志了事，需要保障最终一致性：

```
写入 MySQL 失败
  └─ 先写入 Redis 缓存（作为临时存储）[未实现]
       └─ 同时投入失败队列（Redis List / 本地磁盘队列）[未实现]
            └─ 后台 Worker 循环消费失败队列
                 └─ 重试写入 MySQL（指数退避，最大 10 次）[未实现]
                      └─ 超过重试上限 → 告警 + 人工介入
```

各类写入的优先级和处理策略：

| 写入类型 | 丢失影响 | 临时缓存 | 失败队列 |
|---------|---------|---------|---------|
| 用户消息 / 会话记录 | 高，用户数据丢失 | Redis | ✅ 必须 |
| Memory（profile/facts） | 中，下次对话无记忆 | Redis | ✅ 必须 |
| 账单写入（add_bill） | 高，财务数据不一致 | Redis | ✅ 必须 |
| Memory Determiner 抽取结果 | 低，本轮 Memory 不更新 | — | 可选，Warn 日志即可 |

### 8.3 存储读取失败

读取失败的降级策略分两类：

**有缓存可兜底的：**
- Redis 不可用 → Memory/Session 降级读 MySQL（已实现）
- MySQL 不可用 → Memory 仅读 Redis 缓存，Session 降级为空历史（已实现）
- Redis + MySQL 均不可用 → 空 Memory / 空历史，功能降级但不崩溃（已实现）

**无缓存的（冷数据）：**
- 账单查询（query_bills）MySQL 不可用 → 返回明确错误提示，不 mock 假数据（告知用户稍后重试）[未实现，当前直接报错]

### 8.4 降级等级

| 等级 | 触发条件 | 影响 | 恢复方式 |
|------|----------|------|---------|
| L0 | 正常 | 完整功能 | — |
| L1 | Redis 延迟 >100ms | Memory/Session 读 MySQL，轻微延迟增加 | Redis 自动恢复 |
| L2 | Redis 不可用 | Memory/Session 读 MySQL，写入走失败队列 | Redis 恢复后队列消费 |
| L3 | skill-server 不可用 | 无工具调用，纯对话模式 | skill-server 重启 |
| L4 | 主 LLM 超时/限流 | 切换备用模型，延迟略增 | 主模型恢复后自动切回 |
| L5 | 主备 LLM 均不可用 | Mock 关键词回复兜底 | 人工介入，告警 |
| L6 | MySQL 不可用 | 读降级 Redis；写入失败队列，MySQL 恢复后补偿 | MySQL 恢复后自动消费队列 |

## 9. 安全设计

### 9.1 输入安全

|---------|------|

| 银行卡 | 前6后4 |
