### 1.2 组件职责

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| **API Gateway** | 路由分发、限流、CORS | gorilla/mux + 中间件 |
| **Agent Runtime** | 对话编排、Skill选择、流式响应 | internal/agent |
| **Skill Registry** | Skill注册、匹配、管理 | internal/skill |
| **Skill Loader** | .so插件加载、热更新 | internal/plugin |
| **Memory Service** | 用户记忆管理、Token预算 | internal/memory |
| **Session Manager** | 会话状态管理 | internal/session |
| **Tool Adapter** | 工具调用适配 | internal/plugin/adapter |
| **业务API** | 记账、查询、预算等 | skills/financial |
| **存储层** | Redis会话/MySQL持久化 | redis/go-redis + gorm |

## 3. 数据模型

### 3.1 Session Schema

```go
type Session struct {
    ID          string                 `json:"id"`           // sess_1234567890
    UserID      string                 `json:"user_id"`      // 用户ID
    Messages    []map[string]interface{} `json:"messages"`    // 对话消息
    CreatedAt   string                 `json:"created_at"`   // RFC3339
    LastUpdated string                 `json:"last_updated"` // RFC3339
    ExpiresAt   string                 `json:"expires_at"`   // RFC3339
    Data        map[string]interface{} `json:"data,omitempty"`
}
```

### 3.2 Message Schema

```go
type Message struct {
    ID        string      `json:"id"`         // 消息ID

    Timestamp time.Time   `json:"timestamp"`  // 消息时间
    Metadata  interface{} `json:"metadata,omitempty"`
}
```

type Memory struct {

## 4. 并发与状态管理

### 4.1 会话锁机制

**设计原则：**
- 单会话串行处理，保证消息顺序
- 多会话并行，提高吞吐量
- 锁粒度为会话级别，减少竞争

### 4.2 限流策略

**限流维度：**
| 维度 | 限制 | 目的 |
|------|------|------|
| IP | 100/min | 防DDoS |
| 用户 | 30/min | 保护用户体验 |
| 会话 | 5/s | 防刷屏 |


### 4.3 并发扩容路径

```
阶段1：30并发（当前）
├── 单机部署
├── Redis单节点
└── MySQL单节点

阶段2：60并发（灰度）
├── 2实例 + Nginx负载均衡
├── Redis cluster模式
├── MySQL读写分离
└── 连接池扩容至100

阶段3：100并发（全量）
├── 3实例 + K8s HPA
├── Redis Cluster (3主3从)
├── MySQL主从集群
├── CDN静态资源
└── Prometheus监控告警
```

## 5. 延迟预算分解

### 5.1 端到端延迟目标

| 指标 | 目标 | P95 |
|------|------|-----|
| 首token延迟 | ≤ 1.5s | 2s |
| 整轮响应 | ≤ 4s | 5s |

### 5.2 延迟分解

```
端到端 5s
├── 鉴权 & 限流: 50ms (1%)
├── Session加载: 100ms (2%)
├── Memory读取: 150ms (3%)
├── Prompt组装: 100ms (2%)
├── LLM调用: 3000ms (60%)
│   ├── 网络往返: 200ms
│   ├── 模型推理: 2500ms
│   └── 流式首token: 300ms
├── 流式回写: 1500ms (30%)
│   ├── 网络传输: 500ms
│   └── 客户端渲染: 1000ms
└── Session保存: 100ms (2%)
```

### 5.3 Prompt Caching策略

**缓存策略：**
| 层级 | 缓存内容 | TTL | 命中率 |
|------|----------|-----|--------|
| L1 (内存) | System Prompt | 1h | 80% |
| L2 (Redis) | Skill Context | 10min | 60% |
| L3 (CDN) | 静态模板 | 24h | 95% |

**优化效果：**
- 减少30%的Token发送量
- 降低20%的LLM调用延迟
- 节省15%的API成本

## 6. 容量与成本估算

### 6.1 1万DAU场景分析

**用户行为假设：**
- 日活：10,000
- 日均对话：3次
- 平均轮次：5轮
- 平均Token：200输入 + 400输出

### 6.2 Token用量估算

```
日Token量 = 10,000 × 3 × 5 × (200 + 400) = 90,000,000 tokens
月Token量 = 90M × 30 = 2,700,000,000 tokens
```

### 6.3 模型月成本估算（CNY）

#### 6.3.1 1万DAU月成本方案（按2.7B tokens计算）

| 方案 | 主模型 | 月Token成本 | 基础设施 | 月成本(CNY) | 月成本(USD) |
|------|--------|-------------|----------|-------------|-------------|
| **方案A（推荐）** | DeepSeek V3 | ¥4,050 | ¥200 | **¥4,250** | **$590** |
| **方案B** | GLM-4-Flash | ¥3,375 | ¥200 | **¥3,575** | **$497** |
| **方案C** | Qwen-Turbo | ¥16,200 | ¥200 | **¥16,400** | **$2,278** |

#### 6.3.2 方案推荐

**推荐方案：DeepSeek V3 + GLM-4-Flash 分级策略**

| 场景 | 模型选择 | 占比 | 理由 |
|------|----------|------|------|
| 简单对话（闲聊） | GLM-4-Flash | 40% | 成本最低，响应快 |
| 工具调用/指令执行 | DeepSeek V3 | 40% | 推理能力强，准确率高 |
| 复杂推理/长文本 | DeepSeek V3 | 20% | 128K上下文支持 |

**综合成本：**
```
加权输入成本 = 0.4×0.5 + 0.6×0.5 = ¥0.5/百万tokens
加权输出成本 = 0.4×1.0 + 0.6×2.0 = ¥1.6/百万tokens
月成本 = 1.35B×0.5/百万 + 1.35B×1.6/百万 + ¥200基础设施
      = ¥675 + ¥2,160 + ¥200
      = ¥3,035 ≈ $422/月
```

## 7. 故障预案

### 7.1 LLM限流

**兜底方案：**
1. 降级模型
2. 请求重试（指数退避）

### 7.2 LLM超时
**兜底方案：**
1. 超时后返回部分结果
2. 使用缓存的相似历史回复
3. 返回友好错误提示

### 7.3 Skill损坏
**兜底方案：**
1. 自动降级到参数提取兜底策略（已实现）
2. 从备份恢复Skill
3. 返回"该功能暂时不可用"

### 7.4 Memory DB宕机

**兜底方案：**
1. 使用内存Fallback（当前对话保留）
2. 降低Token Budget（减少历史上下文）
3. 记录到本地日志，恢复后同步

### 7.5 故障降级等级

| 等级 | 触发条件 | 处理方式 |
|------|----------|----------|
| L0 | 正常 | 完整功能 |
| L1 | Redis延迟>100ms | 使用本地缓存 |
| L2 | Redis不可用 | 内存Fallback，无历史 |
| L3 | LLM超时/限流 | Mock响应兜底 |
| L4 | MySQL不可用 | 仅Redis存储 |

## 8. 安全设计

### 8.1 鉴权机制

**安全措施：**
- JWT Token签名验证
- Token过期检查（1小时）
- Refresh Token机制
- HTTPS强制传输

### 8.2 PII脱敏

**脱敏规则：**
| 数据类型 | 脱敏方式 | 示例 |
|---------|---------|------|
| 手机号 | 前3后4 | 138****1234 |
| 邮箱 | 用户名 | user***@domain.com |
| 身份证 | 前6后4 | 110101********1234 |
| 银行卡 | 前6后4 | 622202********1234 |

### 8.3 Prompt注入防护

**防护策略：**
1. 输入过滤：检测注入模式
2. 系统Prompt隔离：System与User分离
3. 输出过滤：敏感信息检测
4. 日志脱敏：不记录原始Prompt

### 8.4 Skill沙箱

**沙箱策略：**
- Go Plugin加载器隔离
- 执行超时限制（5s）
- 内存使用限制（100MB）
- 禁止系统调用
